# Architecture Review & Upgrade Plan: Mini-Wiki 2.0

**To:** [Insert Assistant's Name]
**From:** Hussam
**Date:** May 8, 2026
**Subject:** Transitioning from Semantic RAG to Agentic RAG for Structured Data

### Executive Summary

Mini-wiki has a fantastic foundational architecture. The Go Bubbletea TUI and the local Python worker are highly effective for offline execution. However, during testing, we hit severe performance bottlenecks (e.g., `/rank` taking ~3.8 hours for 92K rows) and accuracy issues.

This is because the current pipeline relies on **Semantic/Linear RAG** for structured data (CSVs). Moving forward, we need to transition the tool to an **Agentic RAG** architecture, where the LLM writes code to filter data rather than reading every row sequentially.

Please review the following three phases to upgrade the architecture.

---

### Phase 1: The Core Fix – Agentic Filtering (The `/rank` Command)

Currently, the Python worker is likely iterating over data chunks to score them row-by-row. Asking an 8B model to grade 92,000 rows sequentially is causing the 3.8-hour latency.

**The Solution:** We must shift from "scoring rows" to "generating scripts."

1. **Schema Extraction:** Instead of passing raw data to the LLM, extract only the column headers and the first 3 rows of the active dataset.
2. **Code Generation:** Pass this schema to the LLM with a prompt asking it to act as a Data Scientist:
* *"Here is the schema of a Pandas DataFrame. Write a Python script to filter out rows that are irrelevant to [Research Topic]. Output only the Python code."*


3. **Local Execution:** Use the Python worker to execute (`exec()` or `eval()`) that generated script locally on the dataset.
*Result:* We reduce 92,000 LLM calls down to **1 LLM call**, reducing `/rank` time from hours to seconds.

---

### Phase 2: Pipeline & UI Stability (Go / VRAM Optimization)

Because we are working within a strict **8GB VRAM** budget on an RTX 5060, the Go application (`main.go` / `modelmgr`) needs tighter control over the hardware.

**1. Prevent VRAM Thrashing:**
Currently, switching between models (e.g., `llama3.1` to a coder model) can cause the system to freeze if VRAM isn't cleared.

* **Action:** Before the Go app triggers a heavy Python task, send an API request to Ollama to forcefully unload the active chat model (set `keep_alive` to `0`). Clear the VRAM completely before spinning up the heavy analytical model.

**2. Remove/Adjust "Context Compaction":**
Pausing the UI for 30 seconds to summarize old chat context ruins the offline experience.

* **Action:** Modern models have massive context windows (128k+). Disable aggressive context summarization and rely on Ollama's native KV-cache shifting. If we must summarize to save RAM, route it to a tiny background model (like a 1.5B model) so the main thread doesn't block.

**3. Implement a Strict JSON Bridge for `/chart`:**
If the LLM writes a Python script that analyzes our data, it cannot return the answer as conversational text, or the Go `chart.go` parser will break.

* **Action:** Enforce strict JSON output from the Python worker. When returning data to the TUI, it must match a schema the Go app expects, for example:
```json
{
  "chart_type": "bar",
  "labels": ["Passed", "Failed"],
  "values": [85, 12]
}

```



---

### Phase 3: Update Local Model Roster (`setup.sh`)

Standard chat models are too generic for our new Agentic architecture. Please update the `setup.sh` and model logic to default to these specialist models in the 8B class:

import json
import logging
import pandas as pd
from rag_worker.querier import call_llm

logger = logging.getLogger(__name__)

def agentic_rank(csv_path: str, topic: str, llm_model: str, ollama_base: str) -> dict:
    """Uses an LLM to write and execute a Pandas script to filter a dataset."""
    try:
        # 1. Load data and extract schema (Do NOT send the whole file)
        df = pd.read_csv(csv_path)
        schema = {col: str(dtype) for col, dtype in df.dtypes.items()}
        sample_rows = df.head(3).to_dict(orient="records")
        
        # 2. Prompt the Coder Model
        system_prompt = (
            "You are an expert Data Scientist. You write flawless Python Pandas code. "
            "Never explain the code. Never use markdown formatting. ONLY output raw Python code."
        )
        
        user_prompt = f"""
        I have a Pandas DataFrame named `df`.
        Schema: {schema}
        Sample Data: {sample_rows}
        
        Task: Write a Python function named `filter_data(df)` that returns a new DataFrame containing 
        ONLY the rows that are highly relevant to the research topic: "{topic}".
        Include a new column called "relevance_score" (1 to 100) based on your logic.
        """
        
        # 3. Get the code from Ollama
        logger.warning(f"Asking {llm_model} to write Pandas script for topic: {topic}")
        generated_code = call_llm(model=llm_model, system=system_prompt, user_message=user_prompt, base_url=ollama_base)
        
        # Strip markdown if the LLM disobeys
        generated_code = generated_code.replace("```python", "").replace("
```", "").strip()
        
        # 4. Safe Execution Sandbox
        local_env = {"pd": pd}
        exec(generated_code, local_env)  # Load the generated function into memory
        
        filter_function = local_env.get("filter_data")
        if not filter_function:
            raise ValueError("The LLM did not generate a 'filter_data' function.")
            
        # 5. Execute on the 92,000 rows (Takes ~0.1 seconds)
        filtered_df = filter_function(df)
        
        # 6. Return as JSON for the Go TUI
        result_json = filtered_df.to_dict(orient="records")
        return {"type": "rank_done", "rows_kept": len(filtered_df), "data": result_json}

    except Exception as e:
        logger.exception("Agentic Rank Failed")
        return {"type": "error", "message": f"Agentic filter failed: {str(e)}"}
